# Vending Machine Parser — отчёт по проекту

**Назначение системы.** Программный комплекс компьютерного зрения,
принимающий на вход фотографию торгового автомата, локализующий сам автомат,
выпрямляющий перспективу его витрины, обнаруживающий каждый товар на полках,
сопоставляющий каждый товар с эталонной галереей образцов и формирующий на
выходе структурированную таблицу (строки × столбцы, одно наименование товара
в каждой ячейке). Поверх backend-сервиса на FastAPI развёрнуто веб-приложение
с захватом изображения через камеру устройства, что позволяет использовать
систему непосредственно со смартфона.

Настоящий документ оформлен как отчёт о проделанной работе и содержит:
описание архитектуры конвейера обработки, методику формирования обучающих
выборок, перечень адаптированных предобученных моделей, описание
веб-приложения и инструкцию по развёртыванию проекта.

## 1. Архитектура конвейера обработки (pipeline)

Конвейер (`pipeline.py`) реализован как последовательность из семи
специализированных моделей и алгоритмических блоков, каждый из которых решает
узкую подзадачу. Результат каждого этапа подаётся на вход следующему
(обрезка/перепроецирование изображения), вследствие чего ошибки накапливаются
кумулятивно — это и есть основная причина, по которой архитектура построена на
наборе небольших специализированных моделей, а не на единой сквозной сети:
так проще локализовать и устранять источник ошибки на конкретном этапе.

Этапы обработки выполняются последовательно:

1. **Детектор торговых автоматов** — YOLOv10n (детекция объектов одного
   класса). Находит на исходной фотографии торговый автомат (или несколько) и
   возвращает ограничивающий прямоугольник (bounding box) для каждого.
2. **Классификатор моделей автоматов** — YOLO26n-cls, применяется к
   вырезанному изображению автомата и определяет его конкретную физическую
   модель. Результат классификации определяет ожидаемое максимальное число
   строк и столбцов сетки (см. словарь `MACHINE_CLASSES` в `shared.py`).
3. **Сегментатор витринного окна** — YOLO26n-seg формирует маску области
   витрины, которую модуль `window_segmentator.py` далее редуцирует до чёткого
   четырёхугольника классическими методами компьютерного зрения:
   морфологическое замыкание/размыкание → детектор границ Канни → выделение
   наибольшего контура → бинарный поиск по параметру epsilon функции
   `approxPolyDP` до получения ровно четырёх вершин (при неудаче — резервный
   алгоритм редукции выпуклой оболочки по критерию минимальной потери площади).
   Полученный четырёхугольник используется для перспективного преобразования
   витрины в выпрямленное, видимое сверху изображение
   (`grid_helper.warp_image`).
4. **Детектор товаров** — YOLO26s-obb (детекция с ориентированными
   ограничивающими прямоугольниками, OBB), запускается **только внутри
   ограничивающего прямоугольника выпрямленного витринного окна**: функции
   `grid_helper.mask_to_polygon`/`offset_obb` вырезают эту область, обнуляя всё
   за её пределами перед детекцией, а затем переносят результаты обратно в
   координаты исходного фото. Использование OBB обеспечивает точные границы
   объектов даже при съёмке под углом.
4b. **Объединение OBB (OBB merge)** — `grid_helper.merge_overlapping_items`:
    объединение через структуру union-find по попарному повёрнутому IoU
    (`ITEM_MERGE_IOU`), а также предварительный проход по «вложенности»
    (`ITEM_MERGE_CONTAINMENT`), отбрасывающий прямоугольники, почти целиком
    покрытые более крупным (дублирующиеся под-детекции).
5. **Построитель сетки (grid builder)** — `grid_helper.build_grid`,
   реализован как чисто геометрический алгоритм без применения моделей
   машинного обучения: вершины OBB каждого товара перепроецируются в
   координаты выпрямленного изображения витрины, товары группируются в строки
   по близости нижней координаты Y (порог — 5% от диапазона Y внутри строки),
   характерная «единичная ширина» столбца для каждой строки вычисляется как
   10–25-й перцентиль расстояний между центрами соседних товаров, каждому
   товару присваивается `floor(ширина / единичная_ширина)` столбцовых позиций,
   после чего границы каждой строки выравниваются по эталонной строке методом
   динамического программирования (DP).
6. **Классификатор товаров** — энкодер эмбеддингов на основе ConvNeXt-Tiny
   (дообученный с использованием ArcFace-головы) формирует векторное
   представление для каждой ячейки сетки; поиск ближайшего соседа по
   косинусному сходству относительно галереи усреднённых эмбеддингов по
   классам (`item_classification.py` / класс `ProductBank`) возвращает
   наименование наиболее похожего товара и оценку достоверности (similarity
   score). Фрагмент для каждой ячейки вырезается напрямую из *исходной*
   фотографии функцией `crop_obb_rotated` — изображение разворачивается по
   углу наклона OBB, приводится к горизонтальной (landscape) ориентации и
   дополняется до квадрата чёрным фоном (так же, как формируются обучающие
   вырезки), после чего сравнивается с галереей.

Все числовые пороги этого конвейера — confidence/IoU детекторов, коэффициенты
IoU/containment для объединения OBB, а также `imgsz` каждой модели YOLO —
собраны в едином dataclass `Config` (`shared.py`) и сохраняются в
`config.json`. Их можно настраивать в реальном времени, на конкретной
фотографии, с помощью `scripts/tune_item_detector.py` (интерфейс со
слайдерами, который при каждом изменении заново запускает детекцию и
объединение OBB и сохраняет принятые значения в `config.json`) — без
редактирования исходного кода.

Итоговым результатом является объект `GridResult` (число строк, число
столбцов, наименование товара и оценка сходства для каждой ячейки), который
может быть отрисован обратно на исходной фотографии — с контуром витринного
окна, рамками отдельных товаров и линиями/подписями сетки (`render_overlay`,
через обратное перспективное преобразование `warp_image_inv`) — либо
экспортирован в виде таблицы Markdown (`build_markdown_table`); именно в этом
виде результат возвращается клиенту через API и отображается в
веб-приложении.

## 2. Формирование собственных (custom) обучающих выборок

Три из пяти моделей конвейера (детектор автоматов, классификатор
автоматов, сегментатор витрины), а также эталонная галерея товаров обучены на
**собственных, вручную собранных данных**, поскольку готовых публичных
датасетов с фотографиями именно этих моделей торговых автоматов и размеченными
витринами не существует.

Эти выборки расположены в каталогах `datasets/` и `gallery/` и были
сформированы при помощи вспомогательных инструментов из каталога `scripts/`:

- **`scripts/image_labeler.py`** — интерактивный инструмент разметки для
  задачи классификации на базе OpenCV. Последовательно показывает изображения
  из заданной папки, позволяет одним нажатием клавиши (`1`–`9`) отнести каждое
  изображение к пронумерованному классу, создавать новые классы «на лету»,
  помечать непригодные изображения для удаления (с проверкой по хэшу SHA-256,
  исключающей повторное появление уже удалённых файлов), а также приводит
  изображения произвольных пропорций к квадратной форме перед сохранением —
  при этом поле `nc` в YAML-файле датасета автоматически синхронизируется с
  фактическим числом классов. Использован для построения выборки
  `vending_machine_classification`.
- **`scripts/split_dataset.py`** — разбивает размеченную выборку на
  обучающую и валидационную части, приводя каждое изображение к квадратному
  холсту (центрирование, чёрный фон) и масштабируя его — то есть применяя ту
  же предобработку, которую модели классификации/эмбеддинга ожидают на этапе
  инференса, что обеспечивает идентичность путей обучения и эксплуатации.
- **`scripts/verify_and_resave_images.py`** — многопроцессный проход на базе
  OpenCV, который перекодирует каждое изображение датасета (исправляя «на
  месте» усечённые/повреждённые байты JPEG) и активно удаляет любое
  изображение — вместе с соответствующим файлом разметки — которое не
  удаётся загрузить или у которого неправдоподобные размеры/число каналов.
  Заменяет более ранний инструмент на базе Pillow `delete_trash_images.py`,
  который лишь сообщал о проблемах, но не исправлял их.
- **`scripts/gallery_labeler.py`** — основной инструмент построения и
  расширения галереи товаров. Запускает *тот же* метод
  `Pipeline.detect(..., classify=False)`, что используется в production
  (детекция автомата → классификация → сегментация витрины → детекция
  товаров с маской по витринному окну), на папке исходных фотографий,
  показывает каждый обнаруженный фрагмент рядом с панелью нечёткого
  поиска/топ-5 предположений и позволяет принять или исправить метку парой
  нажатий клавиш. Эмбеддинги обновляются **инкрементально** —
  `ProductBank.recompute_class()` пересчитывает только те классы, которые
  только что получили новые образцы, — поэтому галерея может расти без
  полной перестройки. Инструмент также выполняет при запуске проверку
  синхронизации между папками `gallery/<class>/` и сохранённой матрицей
  эмбеддингов, а если эмбеддинг только размеченного фрагмента оказывается
  выбросом относительно своего класса (более чем в 2 раза превышает среднее
  внутриклассовое расстояние по всем классам), показывает диалог
  подтверждения — защита от случайных опечаток в метках, которые незаметно
  искажают усреднённый эмбеддинг класса.

Собственные датасеты начинались небольшими и заметно выросли за несколько
раундов съёмки на телефон и разметки описанными выше инструментами —
`vending_machine_detection`, `vending_machine_classification` и
`window_segmentation` теперь содержат по несколько сотен размеченных
изображений, разбитых (приблизительно) в соотношении 80/10/10 на
train/val/test, что достаточно для полноценного обучения детектора автоматов,
классификатора автоматов и сегментатора витрины, а не только для быстрой
проверки на жизнеспособность.

![Новое обучающее фото — недавно снято и размечено для датасета детекции торговых автоматов](screenshots/new_training_photo.jpg)

**Эталонная галерея товаров** (`gallery/<наименование_товара>/*.png`) —
небольшой набор эталонных «чистых» фрагментов товаров (например, `cola_03`,
`snickers`, `twix`, `lays`, `mnms`, `pepero`, `bonaqua`, `dobry_cola_05`),
поддерживаемый с помощью описанного выше `scripts/gallery_labeler.py`. Каждый
фрагмент проходит коррекцию поворота и дополняется до квадрата чёрным фоном
функцией `crop_obb_rotated` (той же, что используется на этапе инференса),
формирует эмбеддинг дообученным энкодером ConvNeXt, **усредняется по
классу**, а итоговая матрица сохраняется в файл
`models/tuned/items_classification.npy` — именно эта матрица служит
эталонной базой, с которой метод `ProductBank.lookup()` сравнивает вырезанные
фрагменты на этапе инференса. (Исходный ноутбук `build_library.ipynb`,
выполнявший полную единоразовую перестройку, выведен из эксплуатации в пользу
этого инкрементального инструмента.)

## 3. Адаптация публичных (глобальных) датасетов

Ручная разметка достаточного количества фотографий торговых автоматов для
обучения детектора товаров «с нуля» практически нереализуема, поэтому в
проекте используются два публичных датасета, адаптированные под формат
разметки и предметную область данной задачи:

- **SKU-110K-R** (`datasets/SKU110K_fixed/`) — крупный публичный датасет для
  задачи детекции товаров на торговых полках (плотно расположенные товары на
  полках магазинов), теперь используемый через его **аннотации с повёрнутыми
  прямоугольниками (SKU110K-R)** вместо исходных осе-выровненных CSV.
  `DRN_CVPR2020/rotate_augment.py` генерирует повёрнутые копии каждого
  исходного изображения вместе с соответствующими ориентированными
  прямоугольниками (`rbbox=[cx,cy,w,h,angle]`);
  `scripts/sku110k_r_to_ultralytics_format.py` преобразует их в один файл
  разметки OBB формата Ultralytics на каждое изображение (train/val/test ≈
  57.5k/4.1k/20.6k изображений, все с настоящими ориентированными метками —
  ранее метки были только у *исходных*, не повёрнутых изображений). В
  сочетании с удалением повреждённых изображений
  (`verify_and_resave_images.py`) это даёт **детектору товаров** (YOLO26s-obb,
  обучение — `notebooks/learn_item_detector.ipynb`) большой, разнообразный и
  теперь полностью *размеченный по ориентации* набор примеров «множество
  мелких прямоугольных товаров на полке», что обеспечивает достаточно хороший
  перенос (transfer) на предметную область торговых автоматов, несмотря на то
  что среди исходных изображений нет ни одного торгового автомата.
- **Retail-YU** (`datasets/Retail-YU_reformed/`) — датасет для задачи
  распознавания розничных товаров (~104 тыс. изображений), реорганизованный в
  структуру вида `ImageFolder`/галерея (`train/`, `val/`, `gallery/`). На его
  основе обучается **модель эмбеддингов для классификации товаров**
  (`notebooks/learn_item_classification.ipynb`): магистральная сеть
  ConvNeXt-Tiny, предобученная на ImageNet-22k→1k
  (`timm/convnext_tiny.fb_in22k_ft_in1k`), дообучается с надстроенной
  **ArcFace-головой** (additive angular margin loss) средствами PyTorch
  Lightning — это обеспечивает «стягивание» эмбеддингов одного товара и
  «расталкивание» эмбеддингов разных товаров в пространстве признаков, то есть
  именно то свойство, которое необходимо для последующего поиска по косинусному
  сходству в галерее. Скрипт `extract_crops_simple.py` использовался для
  извлечения вырезок по ограничивающим прямоугольникам из датасетов формата
  детекции при формировании/расширении обучающей выборки для классификации.

Между двумя публичными датасетами (SKU110K-R ≈ 82 тыс. изображений, Retail-YU
≈ 104 тыс. изображений) и собственными выборками, в каждой из которых теперь
по несколько сотен изображений, у всех пяти дообучаемых моделей теперь
достаточно данных для полноценного цикла обучения, а не только для быстрой
проверки концепции.

### Аугментация окклюзии пружиной/шнеком (spring/auger)

Многие реальные торговые автоматы используют вращающиеся пружинные (шнековые)
спирали для выдачи товара, и эти спирали визуально перекрывают часть товара
широкой дугой перед полкой — примеров такого нет ни в одном из публичных
датасетов. `src/python/spring_augment.py` решает эту задачу процедурной
аугментацией: 18 сфотографированных текстур пружин/шнеков на прозрачном фоне
(`datasets/aug_springs/`) при загрузке разворачиваются в полярные координаты,
чтобы один раз найти их «допустимый диапазон дуги» (исключая разрыв, где
спираль не образует полное кольцо), а функция `random_patch()` затем вырезает
случайную под-дугу (180°–270°) из *исходной* (неразвёрнутой) текстуры с
помощью секторной маски `cv2.ellipse` — без ресэмплинга/искажения текстуры.

Аугментация применяется двумя способами: `SpringOcclusionPIL(p=0.3)` — как
трансформация torchvision для обучающих вырезок классификатора товаров, и
`SpringOcclusion` — привязанная к bbox трансформация датасета Ultralytics для
обучающих изображений детектора товаров; вторая выбирает долю (`coverage`)
истинных прямоугольников на изображении и накладывает патч, размер которого
пропорционален каждому целевому прямоугольнику, так что окклюзии попадают на
реальные товары, а не в случайные точки изображения.

![Текстура окклюзии пружиной/шнеком — одна из 18 исходных текстур, используемых `spring_augment.py`](screenshots/spring_texture_1.png)
![Текстура окклюзии пружиной/шнеком — второй пример, демонстрирующий форму витка спирали](screenshots/spring_texture_2.png)

Кратко: публичные датасеты детекции товаров на полках и распознавания
розничных товаров обеспечивают визуальный *словарь* предметной области (как
выглядит упакованный товар — в массе и крупным планом); небольшие собственные
выборки обеспечивают *геометрию, специфичную для конкретной задачи* (как
выглядит конкретная модель автомата и его витрина); а процедурная аугментация
пружинами добавляет специфичный для торговых автоматов паттерн окклюзии,
примеров которого нет ни в одном из публичных датасетов.

## 4. Предобученные модели и тонкая настройка (fine-tuning)

Все базовые (предобученные) контрольные точки находятся в каталоге
`models/base/`; дообученные в рамках проекта контрольные точки сохраняются в
`models/tuned/` (пути централизованно прописаны в словаре
`shared.MODEL_PATHES`). Каждой дообученной модели соответствует отдельный
обучающий ноутбук:

| Этап | Базовая модель | Дообученная модель | Ноутбук обучения | Обучающая выборка |
|---|---|---|---|---|
| Детектор автоматов | `yolov10n.pt` | `vending_machine_detect_yolov10n.pt` | `learn_vending_machine_detector.ipynb` | `vending_machine_detection` (собственная, с агрессивной аугментацией Albumentations) |
| Классификатор моделей автоматов | `yolo26n-cls.pt` | `vending_machine_classification_yolo26n-cls.pt` | `learn_vending_machine_classification.ipynb` | `vending_machine_classification` (собственная) |
| Сегментатор витрины | `yolo26n-seg.pt` | `window_segmentation_yolo26n-seg.pt` | `learn_window_segmentation.ipynb` | `window_segmentation` (собственная; обучение с аугментацией поворотом/отражением/мозаикой + Albumentations) |
| Детектор товаров | `yolo26s-obb.pt` | `items_detect_yolo26s-obb.pt` | `learn_item_detector.ipynb` | `SKU110K_fixed` (SKU110K-R, OBB-метки для исходных и повёрнутых изображений, плюс аугментация окклюзией пружиной/шнеком) |
| Энкодер эмбеддингов товаров | ConvNeXt-Tiny `timm/convnext_tiny.fb_in22k_ft_in1k` (ImageNet) | `items_classification_convnext_tiny.fb_in22k_ft_in1k.pt` (+ ArcFace-голова) | `learn_item_classification.ipynb` | `Retail-YU_reformed` (публичная, переформатирована, с поворотом на 0/90/180/270° + аугментацией окклюзией пружиной) |
| Галерея товаров | — | `items_classification.npy` (+ `items_classification_spread.json`, нормализованные усреднённые эмбеддинги по классам) | `scripts/gallery_labeler.py` | `gallery/` (собственные эталонные вырезки, скорректированные по повороту и дополненные чёрным фоном через `crop_obb_rotated`) |

В каталоге `models/base/` также присутствуют дополнительные контрольные точки
(`yolo11n.pt`, `yolo26n.pt`), сохранённые для экспериментов и в текущей
версии конвейера не задействованные.

Каждая дообученная модель и эмбеддинги галереи прошли как минимум один цикл
переобучения/перестройки после того, как таблица выше впервые
стабилизировалась, и каждый такой цикл устранял реальное расхождение между
обучением и инференсом: центроиды галереи эмбеддингов теперь
L2-нормализуются перед поиском по косинусному сходству, вырезки товаров
теперь развёрнуты по углу наклона (`crop_obb_rotated`) вместо
осе-выровненного прямоугольника вокруг повёрнутого OBB, а обучение и
инференс теперь одинаково дополняют вырезки до квадрата **чёрным** фоном
(как делает `_pad_square` в `ProductBank`), а не цветом, усреднённым по краям
изображения, как было в более ранней версии.

## 5. Веб-приложение

Каталог `web/` содержит компактное одностраничное приложение на базе Vite +
TypeScript (без фреймворков и библиотек управления состоянием — только DOM),
которое выполняет следующие действия:

1. Запрашивает доступ к камере устройства
   (`navigator.mediaDevices.getUserMedia`, с предпочтением задней камеры и
   запросом разрешения вплоть до 4K — для получения кадров, достаточно чётких
   для последующей детекции), отображает живое превью и текстовую подсказку о
   необходимости полностью разместить автомат в кадре.
2. Захватывает текущий кадр в canvas и отправляет его в формате JPEG методом
   `multipart/form-data` на конечную точку backend `POST /recognize`.
3. Отображает полученный ответ: выпрямленное изображение витрины с
   нанесёнными линиями сетки и подписями (передаётся в виде base64-строки
   JPEG), а также HTML-таблицу, построенную на основе структурированных данных
   по каждой ячейке — наименование распознанного товара, оценка сходства и
   эталонная миниатюра, загружаемая через `GET /products/{name}/image`.
   У таблицы может быть больше десятка столбцов (по одному на каждый столбец
   сетки, например C0–C11 для широких автоматов), поэтому она оборачивается в
   элемент `.table-wrapper` со свойством `overflow-x: auto` — на узких/
   мобильных экранах таблица прокручивается по горизонтали внутри своей
   секции с рамкой, а не выходит за пределы страницы.

Приложение **обслуживается тем же процессом FastAPI/uvicorn**
(`api.py` монтирует каталог `web/dist` как статические файлы), благодаря чему
frontend и backend работают в рамках одного origin — настройка CORS не
требуется, все запросы используют относительные пути. Доступ ограничивается
по bearer-токену (файл `tokens.yaml`, токен сохраняется на стороне клиента в
`localStorage` / передаётся через query-параметр `?token=`). Подробные
инструкции по сборке и запуску в режиме разработки см. в
[`web/README.md`](../web/README.md).

## Текущие возможности системы (резюме)

- Обнаружение торговых автоматов на произвольных фотографиях и определение
  их конкретной модели.
- Сегментация и выпрямление перспективы витринного окна.
- Детекция отдельных товаров с использованием ориентированных ограничивающих
  прямоугольников (OBB), в том числе при плотном расположении.
- Восстановление структуры полок в виде сетки строк и столбцов чисто
  геометрическим методом (без применения моделей машинного обучения на этом
  этапе).
- Сопоставление каждого обнаруженного товара с небольшой эталонной галереей
  по сходству эмбеддингов с указанием оценки достоверности для каждой ячейки.
- Предоставление всего перечисленного функционала через защищённый токеном
  REST API и адаптированное под мобильные устройства веб-приложение с камерой;
  результат может быть представлен как в виде размеченного изображения, так
  и в виде таблицы Markdown.

## Дальнейшие направления работы

Все пять моделей с тех пор прошли полный цикл переобучения на описанных выше
датасетах и аугментациях (окклюзия пружиной/шнеком, ориентированные метки
SKU-110K-R, аугментация поворотом, а также исправления расхождений между
обучением и инференсом — нормализация центроидов галереи и единообразное
дополнение `crop_obb_rotated` чёрным фоном), и результаты распознавания на
реальных фотографиях теперь хорошие. Поскольку точность моделей больше не
является основным ограничением, оставшаяся работа касается покрытия галереи,
корректности сетки и времени вычислений:

- **Расширение галереи товаров.** Точность распознавания теперь в основном
  ограничена тем, для какого количества реальных товаров в `gallery/` есть
  эталонные вырезки, а не самой моделью эмбеддингов. `scripts/gallery_labeler.py`
  делает добавление новых товаров инкрементальным — достаточно разметить
  несколько вырезок нового товара, инструмент пересчитывает центроид этого
  класса, и он сразу становится доступен для `ProductBank.lookup()`.
  Естественный следующий шаг — просто запускать инструмент на новых
  фотографиях, расширяя покрытие по SKU (и попутно ввести соглашение об
  именовании для эталонных изображений, которые сопоставление должно *не*
  считать валидным товаром — например, пустые/перекрытые кадры, которые не
  должны «притягивать» ячейку к какому-либо классу).
- **Обработка пустых слотов на полке в сетке.** `grid_helper.build_grid`
  сейчас строит строки/столбцы исключительно на основе *обнаруженных* товаров,
  поэтому пустой слот (на полке нет товара) не имеет соответствующего OBB и
  либо сдвигает нумерацию столбцов для всего, что идёт после него, либо
  незаметно выпадает из сетки. Детектор товаров/построитель сетки нужно
  настроить так, чтобы распознавать такие пропуски и либо скрывать их в
  отображаемой таблице, сохраняя при этом за ними позицию по строке/столбцу
  (чтобы остальная часть сетки оставалась корректно выровненной), либо
  показывать их как явные «пустые» ячейки.
- **Оптимизация времени вычислений конвейера.** `pipeline.py` сейчас
  запускает свои модели строго последовательно на каждое фото, а
  классификатор товаров — по одному разу на каждую обнаруженную ячейку.
  Вероятные направления оптимизации: объединение поячейковых запросов
  эмбеддингов в один проход (batch), кэширование/повторное использование
  промежуточных результатов (например, выпрямленного изображения витрины)
  между повторными запросами, а также проверка, действительно ли детектору
  товаров `yolo26s-obb` нужен текущий размер входа `736px` или он может
  работать с меньшим разрешением без потери полноты обнаружения (recall).

![Иллюстрация типичного сбоя — meme-картинка из рабочих материалов, добавлена для разрядки](screenshots/failure%20case.png)
