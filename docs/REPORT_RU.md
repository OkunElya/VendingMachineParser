# Vending Machine Parser — отчёт по проекту

**Назначение системы.** Программный комплекс компьютерного зрения,
принимающий на вход фотографию торгового автомата, локализующий сам автомат,
выпрямляющий перспективу его витрины, обнаруживающий каждый товар на полках,
сопоставляющий каждый товар с эталонной галереей образцов и формирующий на
выходе структурированную таблицу (строки × столбцы, одно наименование товара
в каждой ячейке). Поверх backend-сервиса на FastAPI развёрнуто веб-приложение
с захватом изображения через камеру устройства, что позволяет использовать
систему непосредственно со смартфона.

Настоящий документ оформлен как отчёт о проделанной работе и содержит:
описание архитектуры конвейера обработки, методику формирования обучающих
выборок, перечень адаптированных предобученных моделей, описание
веб-приложения и инструкцию по развёртыванию проекта.

---

## 1. Архитектура конвейера обработки (pipeline)

Конвейер (`pipeline.py`) реализован как последовательность из шести
специализированных моделей и алгоритмических блоков, каждый из которых решает
узкую подзадачу. Результат каждого этапа подаётся на вход следующему
(обрезка/перепроецирование изображения), вследствие чего ошибки накапливаются
кумулятивно — это и есть основная причина, по которой архитектура построена на
наборе небольших специализированных моделей, а не на единой сквозной сети:
так проще локализовать и устранять источник ошибки на конкретном этапе.

Этапы обработки выполняются последовательно:

1. **Детектор торговых автоматов** — YOLOv10n (детекция объектов одного
   класса). Находит на исходной фотографии торговый автомат (или несколько) и
   возвращает ограничивающий прямоугольник (bounding box) для каждого.
2. **Классификатор моделей автоматов** — YOLO26n-cls, применяется к
   вырезанному изображению автомата и определяет его конкретную физическую
   модель. Результат классификации определяет ожидаемое максимальное число
   строк и столбцов сетки (см. словарь `MACHINE_CLASSES` в `shared.py`).
3. **Сегментатор витринного окна** — YOLO26n-seg формирует маску области
   витрины, которую модуль `window_segmentator.py` далее редуцирует до чёткого
   четырёхугольника классическими методами компьютерного зрения:
   морфологическое замыкание/размыкание → детектор границ Канни → выделение
   наибольшего контура → бинарный поиск по параметру epsilon функции
   `approxPolyDP` до получения ровно четырёх вершин (при неудаче — резервный
   алгоритм редукции выпуклой оболочки по критерию минимальной потери площади).
   Полученный четырёхугольник используется для перспективного преобразования
   витрины в выпрямленное, видимое сверху изображение
   (`grid_helper.warp_image`).
4. **Детектор товаров** — YOLO26n-obb (детекция с ориентированными
   ограничивающими прямоугольниками, OBB). Находит каждый товар на полках;
   использование OBB обеспечивает точные границы объектов даже при съёмке под
   углом.
5. **Построитель сетки (grid builder)** — `grid_helper.build_grid`,
   реализован как чисто геометрический алгоритм без применения моделей
   машинного обучения: вершины OBB каждого товара перепроецируются в
   координаты выпрямленного изображения витрины, товары группируются в строки
   по близости нижней координаты Y (порог — 3% от общего диапазона по Y),
   вычисляется характерная «единичная ширина» столбца как среднее по 25%
   самых узких объектов, и каждому товару присваивается
   `floor(ширина / единичная_ширина)` столбцовых позиций.
6. **Классификатор товаров** — энкодер эмбеддингов на основе ConvNeXt-Tiny
   (дообученный с использованием ArcFace-головы) формирует векторное
   представление для каждой вырезанной ячейки сетки; поиск ближайшего соседа
   по косинусному сходству относительно галереи усреднённых эмбеддингов по
   классам (`item_classification.py` / класс `ProductBank`) возвращает
   наименование наиболее похожего товара и оценку достоверности
   (similarity score).

Итоговым результатом является объект `GridResult` (число строк, число
столбцов, наименование товара и оценка сходства для каждой ячейки), который
может быть как отрисован поверх выпрямленного изображения (`render_grid`),
так и экспортирован в виде таблицы Markdown (`build_markdown_table`) — именно
в этом виде результат возвращается клиенту через API и отображается в
веб-приложении.

## 2. Формирование собственных (custom) обучающих выборок

Три из шести компонентов конвейера (детектор автоматов, классификатор
автоматов, сегментатор витрины), а также эталонная галерея товаров обучены на
**собственных, вручную собранных данных**, поскольку готовых публичных
датасетов с фотографиями именно этих моделей торговых автоматов и размеченными
витринами не существует.

Эти выборки расположены в каталогах `datasets/` и `gallery/` и были
сформированы при помощи вспомогательных инструментов из каталога `scripts/`:

- **`scripts/image_labeler.py`** — интерактивный инструмент разметки для
  задачи классификации на базе OpenCV. Последовательно показывает изображения
  из заданной папки, позволяет одним нажатием клавиши (`1`–`9`) отнести каждое
  изображение к пронумерованному классу, создавать новые классы «на лету»,
  помечать непригодные изображения для удаления (с проверкой по хэшу SHA-256,
  исключающей повторное появление уже удалённых файлов), а также приводит
  изображения произвольных пропорций к квадратной форме перед сохранением —
  при этом поле `nc` в YAML-файле датасета автоматически синхронизируется с
  фактическим числом классов. Использован для построения выборки
  `vending_machine_classification`.
- **`scripts/split_dataset.py`** — разбивает размеченную выборку на
  обучающую и валидационную части, приводя каждое изображение к квадратному
  холсту (центрирование, чёрный фон) и масштабируя его — то есть применяя ту
  же предобработку, которую модели классификации/эмбеддинга ожидают на этапе
  инференса, что обеспечивает идентичность путей обучения и эксплуатации.
- **`scripts/delete_trash_images.py`** — выполняет проверку выборки на
  предмет повреждённых/усечённых файлов изображений (глубокая побайтовая
  верификация средствами Pillow, а не только проверка заголовков), что
  предотвращает аварийное завершение длительного процесса обучения из-за
  единственного некорректного файла JPEG.

**Эталонная галерея товаров** (`gallery/<наименование_товара>/*.jpg`) —
небольшой набор эталонных фотографий товаров (например, `cola_03`,
`snickers`, `twix`, `lays`, `mnms`, `pepero`, `bonaqua`, `dobry_cola_05`).
Ноутбук `notebooks/build_library.ipynb` обходит эту директорию, прогоняет
через детектор товаров каждое изображение, вырезает область наибольшего
обнаруженного OBB, формирует эмбеддинг дообученным энкодером ConvNeXt,
**усредняет все эмбеддинги в пределах класса** и сохраняет итоговую матрицу в
файл `models/tuned/items_classification.npy` — именно эта матрица служит
эталонной базой, с которой метод `ProductBank.lookup()` сравнивает вырезанные
фрагменты на этапе инференса.

## 3. Адаптация публичных (глобальных) датасетов

Ручная разметка достаточного количества фотографий торговых автоматов для
обучения детектора товаров «с нуля» практически нереализуема, поэтому в
проекте используются два публичных датасета, адаптированные под формат
разметки и предметную область данной задачи:

- **SKU-110K** (`datasets/SKU110K_fixed/`) — крупный публичный датасет для
  задачи детекции товаров на торговых полках (плотно расположенные товары на
  полках магазинов). Его CSV-аннотации преобразуются в формат разметки OBB
  библиотеки Ultralytics скриптом `scripts/sku_to_ultralytics_format.py`
  (построчное преобразование CSV каждой выборки в текстовые файлы разметки на
  каждое изображение), повреждённые изображения удаляются скриптом
  `delete_trash_images.py`. Этот датасет предоставляет **детектору товаров**
  (YOLO26n-obb, обучение — `notebooks/learn_item_detector.ipynb`) большой и
  разнообразный набор примеров «множество мелких прямоугольных товаров на
  полке», что обеспечивает достаточно хороший перенос (transfer) на предметную
  область торговых автоматов, несмотря на то что среди исходных изображений
  нет ни одного торгового автомата.
- **Retail-YU** (`datasets/Retail-YU_reformed/`) — датасет для задачи
  распознавания розничных товаров, реорганизованный в структуру вида
  `ImageFolder`/галерея (`train/`, `val/`, `gallery/`). На его основе обучается
  **модель эмбеддингов для классификации товаров**
  (`notebooks/learn_item_classification.ipynb`): магистральная сеть
  ConvNeXt-Tiny, предобученная на ImageNet-22k→1k
  (`timm/convnext_tiny.fb_in22k_ft_in1k`), дообучается с надстроенной
  **ArcFace-головой** (additive angular margin loss) средствами PyTorch
  Lightning — это обеспечивает «стягивание» эмбеддингов одного товара и
  «расталкивание» эмбеддингов разных товаров в пространстве признаков, то есть
  именно то свойство, которое необходимо для последующего поиска по косинусному
  сходству в галерее. Скрипт `extract_crops_simple.py` использовался для
  извлечения вырезок по ограничивающим прямоугольникам из датасетов формата
  детекции при формировании/расширении обучающей выборки для классификации.

Кратко: публичные датасеты детекции товаров на полках и распознавания
розничных товаров обеспечивают визуальный *словарь* предметной области (как
выглядит упакованный товар — в массе и крупным планом); небольшие собственные
выборки обеспечивают *геометрию, специфичную для конкретной задачи* (как
выглядит конкретная модель автомата и его витрина).

## 4. Предобученные модели и тонкая настройка (fine-tuning)

Все базовые (предобученные) контрольные точки находятся в каталоге
`models/base/`; дообученные в рамках проекта контрольные точки сохраняются в
`models/tuned/` (пути централизованно прописаны в словаре
`shared.MODEL_PATHES`). Каждой дообученной модели соответствует отдельный
обучающий ноутбук:

| Этап | Базовая модель | Дообученная модель | Ноутбук обучения | Обучающая выборка |
|---|---|---|---|---|
| Детектор автоматов | `yolov10n.pt` | `vending_machine_detect_yolov10n.pt` | `learn_vending_machine_detector.ipynb` | `vending_machine_detection` (собственная) |
| Классификатор моделей автоматов | `yolo26n-cls.pt` | `vending_machine_classification_yolo26n-cls.pt` | `learn_vending_machine_classification.ipynb` | `vending_machine_classification` (собственная) |
| Сегментатор витрины | `yolo26n-seg.pt` | `window_segmentation_yolo26n-seg.pt` | `learn_window_segmentation.ipynb` | `window_segmentation` (собственная; обучение с аугментацией поворотом/отражением/мозаикой) |
| Детектор товаров | `yolo26n-obb.pt` | `items_detect.yolo26n-obb.pt` | `learn_item_detector.ipynb` | `SKU110K_fixed` (публичная, адаптирована под OBB) |
| Энкодер эмбеддингов товаров | ConvNeXt-Tiny `timm/convnext_tiny.fb_in22k_ft_in1k` (ImageNet) | `items_classification_convnext_tiny.fb_in22k_ft_in1k.pt` (+ ArcFace-голова) | `learn_item_classification.ipynb` | `Retail-YU_reformed` (публичная, переформатирована) |
| Галерея товаров | — | `items_classification.npy` (усреднённые эмбеддинги по классам) | `build_library.ipynb` | `gallery/` (собственные эталонные фотографии) |

В каталоге `models/base/` также присутствуют дополнительные контрольные точки
(`yolo11n.pt`, `yolo26n.pt`), сохранённые для экспериментов и в текущей
версии конвейера не задействованные.

## 5. Веб-приложение

Каталог `web/` содержит компактное одностраничное приложение на базе Vite +
TypeScript (без фреймворков и библиотек управления состоянием — только DOM),
которое выполняет следующие действия:

1. Запрашивает доступ к камере устройства
   (`navigator.mediaDevices.getUserMedia`, с предпочтением задней камеры и
   запросом разрешения вплоть до 4K — для получения кадров, достаточно чётких
   для последующей детекции), отображает живое превью и текстовую подсказку о
   необходимости полностью разместить автомат в кадре.
2. Захватывает текущий кадр в canvas и отправляет его в формате JPEG методом
   `multipart/form-data` на конечную точку backend `POST /recognize`.
3. Отображает полученный ответ: выпрямленное изображение витрины с
   нанесёнными линиями сетки и подписями (передаётся в виде base64-строки
   JPEG), а также HTML-таблицу, построенную на основе структурированных данных
   по каждой ячейке — наименование распознанного товара, оценка сходства и
   эталонная миниатюра, загружаемая через `GET /products/{name}/image`.

Приложение **обслуживается тем же процессом FastAPI/uvicorn**
(`api.py` монтирует каталог `web/dist` как статические файлы), благодаря чему
frontend и backend работают в рамках одного origin — настройка CORS не
требуется, все запросы используют относительные пути. Доступ ограничивается
по bearer-токену (файл `tokens.yaml`, токен сохраняется на стороне клиента в
`localStorage` / передаётся через query-параметр `?token=`). Подробные
инструкции по сборке и запуску в режиме разработки см. в
[`web/README.md`](../web/README.md).

## Текущие возможности системы (резюме)

- Обнаружение торговых автоматов на произвольных фотографиях и определение
  их конкретной модели.
- Сегментация и выпрямление перспективы витринного окна.
- Детекция отдельных товаров с использованием ориентированных ограничивающих
  прямоугольников (OBB), в том числе при плотном расположении.
- Восстановление структуры полок в виде сетки строк и столбцов чисто
  геометрическим методом (без применения моделей машинного обучения на этом
  этапе).
- Сопоставление каждого обнаруженного товара с небольшой эталонной галереей
  по сходству эмбеддингов с указанием оценки достоверности для каждой ячейки.
- Предоставление всего перечисленного функционала через защищённый токеном
  REST API и адаптированное под мобильные устройства веб-приложение с камерой;
  результат может быть представлен как в виде размеченного изображения, так
  и в виде таблицы Markdown.

## Инструкция по развёртыванию проекта

### 1. Требования к окружению

- **Python 3.12.11** (проект разрабатывается и тестируется именно на этой
  версии — использование иного патч-релиза 3.12.x, вероятно, допустимо, но
  иная минорная версия не гарантирует совместимости с зафиксированными в
  `requirements.txt` версиями пакетов, в первую очередь — со сборкой `torch`
  с поддержкой CUDA).
- **Node.js и npm** (требуются только при необходимости сборки/изменения
  веб-интерфейса — см. [`web/README.md`](../web/README.md)).
- Для обучения моделей настоятельно рекомендуется наличие GPU с поддержкой
  CUDA; инференс на GPU также заметно быстрее, однако вся система
  работоспособна и на CPU (повсеместно используется конструкция
  `torch.device("cuda" if torch.cuda.is_available() else "cpu")`).

### 2. Клонирование репозитория и создание виртуального окружения

```sh
git clone <ссылка-на-репозиторий>
cd vendingMachineParser

python3.12 -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate

pip install --upgrade pip
pip install -r requirements.txt
```

> Зафиксированные в `requirements.txt` сборки `torch`/`torchvision`/
> `torchaudio` ориентированы на CUDA 12.9 (`+cu129`). При работе только на
> CPU либо с иной версией CUDA необходимо **предварительно** установить
> соответствующую сборку с сайта
> https://pytorch.org/get-started/locally/, после чего выполнить
> `pip install -r requirements.txt`, либо отредактировать указанные три
> строки файла самостоятельно.

### 3. Получение датасетов

Датасеты **не включены** в состав репозитория (см. `datasets/.gitignore`)
ввиду их объёма. Их необходимо загрузить и разместить в каталоге `datasets/`
в соответствии с именами папок, на которые ссылаются обучающие ноутбуки и
файлы `*.yaml`:

| Датасет | Ожидаемый путь | Источник |
|---|---|---|
| Детекция торговых автоматов (собственный) | `datasets/vending_machine_detection/` | будет |
| Классификация моделей автоматов (собственный) | `datasets/vending_machine_classification/` |будет |
| Сегментация витрины (собственный) | `datasets/window_segmentation/` | будет  |
| Эталонная галерея товаров (собственный) | `gallery/` |спаршено |

### 5. Настройка токенов доступа

Скопировать файл-образец и заменить значения-заглушки реальными секретами:

```sh
cp tokens_example.yaml tokens.yaml
```

В файле `tokens.yaml` для каждого клиента (`web-app`, `dev` и т. д.) задать
действительные секретные значения — это bearer-токены, которые клиенты должны
передавать в заголовке `Authorization: Bearer <token>`. Файл `tokens.yaml`
исключён из системы контроля версий (`.gitignore`); реальные токены
недопустимо включать в коммиты.

### 6. Сборка веб-интерфейса (опционально, но рекомендуется)

```sh
cd web
npm install
npm run build      # формирует каталог web/dist/, который обслуживается api.py
cd ..
```

Подробные инструкции по запуску в режиме разработки — в
[`web/README.md`](../web/README.md).

### 7. Запуск backend-сервиса

```sh
python3 api.py
```

Команда запускает uvicorn по адресу `http://0.0.0.0:8004`, монтирует каталог
`web/dist` (при наличии сборки) в корень `/` и предоставляет следующие
конечные точки:

- `POST /recognize` — загрузка фотографии (`multipart/form-data`, поле
  `file`), возвращает результат разбора автомата(ов) в виде структурированной
  сетки.
- `GET /products/{name}/image` — получение эталонной миниатюры товара из
  галереи.

Для использования веб-приложения с камерой следует открыть в современном
браузере адрес `http://localhost:8004/?token=<ваш_токен>` (доступ к камере
требует защищённого соединения — `https://` либо `localhost`).

## Дальнейшие направления работы

- **Расширение собственной обучающей выборки.** Модель сегментации витрины
  на практике довольно часто даёт сбой — нестандартные ракурсы съёмки, блики
  и отражения на стекле, частичные перекрытия и неоднородное освещение
  приводят к ошибкам сегментации, которые далее каскадно приводят к неудачному
  перспективному преобразованию и пропуску автомата при обработке (в этом
  случае `pipeline.py` ограничивается записью в журнал сообщения
  `"failed to segment window"` и переходит к следующему кандидату). Пополнение
  выборки `window_segmentation` (а вероятно, и
  `vending_machine_detection`/`vending_machine_classification`) более
  разнообразными и сложными примерами должно заметно повысить устойчивость
  данного этапа.
- **Дополнительная тонкая настройка модели эмбеддингов.** Модель
  классификации товаров (ConvNeXt-Tiny + ArcFace) в текущем виде обеспечивает
  довольно слабое разделение классов в пространстве эмбеддингов — оценки
  косинусного сходства на реальных сетках нередко близки к нулю, а иногда
  модель указывает неверный товар (см. пример вывода в `workspace/temp.md`,
  где значительная часть ячеек уверенно отнесена не к тому классу — например,
  путаница между `cocacola` и `snickers`). Увеличение объёма и повышение
  качества обучающей выборки, более агрессивная аугментация, приближенная к
  реальным условиям съёмки (ракурс, освещение, частичное перекрытие), и/или
  увеличение числа эпох обучения на `Retail-YU_reformed` (либо на её
  расширенном аналоге) предположительно позволят повысить разделимость классов
  и итоговую точность сопоставления.
